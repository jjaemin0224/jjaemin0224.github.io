[개념] 고프 디자인 패턴(GoF Design Pattern)

1. GoF Design Pattern

고프 디자인 패턴(GoF Design Patterns)은 “Design Patterns : Elements of Resusable Object-Oriented Software”라는 책에서 소개된 23가지 소프트웨어 디자인 패턴의 잡합을 가리키는 용어입니다. 이 책은 에리히 감마, 리처드 헬름, 랄프 존슨, 존 블리시디스라는 저자들에 의해 쓰여진 것으로, 소프트웨어 개발에서 자주 발생하는 문제를 해결하기 위한 열련의 설계 아이디어와 방법을 제시합니다.

디자인 패턴은 소프트웨어 개발자들이 재사용 가능하고 유연한 소프트웨어 시스템을 설계하는 데 도움이 됩니다. 각 패턴은 특정한 설계 문제를 해결하기 위한 구조와 상호작용 방식을 제공하며, 코드의 가독성과 재사용성을 향상시키고 유지보수를 용이하게 만듭니다.

GoF 디자인 패턴은 크게 세 가지 카테고리로 나눌 수 있습니다.

  1. 생성 패턴(Creational Pattern) : 객체의 생성과 관련된 패턴으로, 객체 생성을 추상화하고 유연성을 제공합니다.
  2. 구조 패턴(Structural Pattern) : 클래스와 객체를 조합하여 더 큰 구조를 형성하는 패턴으로, 객체 간의 관계를 정의하고 구성합니다.
  3. 행위 패턴(Behavioral Pattern) : 객체 간의 상호작용과 책임 분배에 관련된 패턴으로, 알고리즘의 분리, 객체 간의 통신, 이벤트 처리 등을 다룹니다.

2. 생성 패턴

  1. 싱글톤(Singleton) 패턴
    • 목적 : 클래스의 인스턴스를 하나만 생성하고, 이를 전역적으로 접근 가능하게 합니다.
    • 구현 : 생성자를 private로 선언하여 직접적인 인스턴스화를 방지하고, 정적 메서드를 통해 유일한 인스턴스에 접근할 수 있게 합니다.
  2. 팩토리 메서드(Factory Method) 패턴
    • 목적 : 객체 생성을 캡슐화하고, 객체의 생성 방식에 대한 유연성을 제공합니다.
    • 구현 : 인터페이스를 정의하고, 서브클래스에서 해당 인터페이스를 구현하여 객체를 생성합니다.
  3. 추상 팩토리(Abstract Factory) 패턴
    • 목적 : 관련된 객체들의 생성을 추상화하고, 객체 그룹을 교체 가능하게 합니다.
    • 구현 : 추상 팩토리 인터페이스를 정의하고, 구체적인 팩토리 클래스에서 해당 인터페이스를 구현하여 연관된 객체들을 생성합니다.
  4. 빌더(Builder) 패턴
    • 목적 : 복잡한 객체의 생성 과정을 추상화하고, 객체 생성 단계를 분리하여 유연하게 객체를 생성합니다.
    • 구현 : 빌더 인터페이스를 정의하고, 빌더 클래스를 사용하여 객체의 생성 과정을 조립합니다.
  5. 프로토타입(Prototype) 패턴
    • 목적 : 객체를 복제하여 생성하고, 기존 객체의 상태를 복사하여 새로운 객체를 생성합니다.
    • 구현 : 객체가 자기 자신을 복제할 수 있는 Clone 메서드를 구현합니다.

생성 패턴은 객체 생성을 다루는 디자인 패턴입니다. 이러한 패턴은 객체의 생성과 초기화를 추상화하고 유연성을 제공하여 객체 생성에 관련된 복잡성을 해결합니다.

3. 구조 패턴

  1. 어댑터(Adapter) 패턴
    • 목적 : 호환되지 않는 인터페이스를 가진 클래스들을 함께 동작할 수 있게 맞추어줍니다.
    • 구현 : 어댑터 클래스를 사용하여 클라이언트와 타깃 클래스 간의 인터페이스를 변환합니다.
  2. 브릿지(Bridge) 패턴
    • 목적 : 추상화와 구현부를 분리하여 각각 독립적으로 확장할 수 있게 합니다.
    • 구현 : 추상화와 구현부를 각각 클래스로 분리하고, 다리 역할의 인터페이스를 통해 서로 연결합니다.
  3. 컴포지트(Composite) 패턴
    • 목적 : 객체들을 트리 구조로 구성하여 단일 객체와 객체 그룹을 동일하게 다룰 수 있게 합니다.
    • 구현 : 객체들을 컴포지트 객체와 리프(leaf) 객체로 구분하고, 컴포지트 객체는 하위 객체들을 관리합니다.
  4. 데코레이터(Decorator) 패턴
    • 목적 : 객체의 기능을 동적으로 확장하거나 변경합니다.
    • 구현 : 데코레이터 클래스를 사용하여 객체에 추가적인 기능을 런타임에 덧붙입니다.
  5. 퍼사드(Facade) 패턴
    • 목적 : 복잡한 서브시스템을 단순화하여 클라이언트에게 단일 인터페이스를 제공합니다.
    • 구현 : 퍼사드 클래스를 사용하여 서브시스템의 인터페이스를 감싸고, 클라이언트에게 단순한 인터페이스를 제공합니다.
  6. 플라이웨이트(Flyweight) 패턴
    • 목적 : 객체들 사이의 공유 가능한 데이터를 최대한 활용하여 메모리 사용을 최적화합니다.
    • 구현 : 객체들을 내부적으로 공유하여 중복 데이터를 최소합니다.
  7. 프록시(Proxy) 패턴
    • 목적 : 접근 제어, 보안, 원격 호출 등을 위한 대리자 객체를 제공합니다.
    • 구현 : 프록시 객체를 사용하여 원본 객체에 대한 접근을 제어하고 추가 기능을 제공합니다.

구조 패턴은 객체들 사이의 구조와 관계를 조직화하고, 이를 통해 유연하고 효율적인 소프트웨어 설계를 돕는 디자인 패턴입니다. 이러한 패턴은 클래스와 객체를 조합하여 더 큰 구조를 형성하고, 객체 간의 상호작용을 간소화하는 방법을 제공합니다.

4. 행위 패턴

  1. 책임 연쇄(Chain of Responsibility) 패턴
    • 목적 : 요청을 처리하는 객체들의 연결된 체인을 구성하여 처리할 수 있는 객체를 찾고, 요청을 처리합니다.
    • 구현 : 각 처리 객체는 요청을 처리할 수 있는지 확인하고, 처리할 수 없는 경우 다음 처리 객체로 전달합니다.
  2. 커맨드(Command) 패턴
    • 목적 : 요청을 객체의 형태로 캡슐화하고, 요청을 매개변수화하여 연산의 실행을 요청자와 수신자 사이에 분리합니다.
    • 구현 : 커맨드 객체는 요청을 수신자에게 전달하고, 수신자는 요청을 실행합니다.
  3. 인터프리터(Interpreter) 패턴
    • 목적 : 언어나 문법을 해석하고, 이를 실행하기 위한 인터프리터를 구현합니다.
    • 구현 : 문법 요소를 표현하는 클래스들을 정의하고, 인터프리터가 문법을 해석하고 실행합니다.
  4. 이터레이터(Iterator) 패턴
    • 목적 : 컬렉션 내부의 요소를 순차적으로 접근하고 조작하는 방법을 제공합니다.
    • 구현 : 이터레이터 객체를 사용하여 컬렉션의 요소에 접근하고, 다음 요소로 이동합니다.
  5. 중재자(Mediator) 패턴
    • 목적 : 객체들 사이의 상호작용을 캡슐화하여 객체 간의 직접적인 상호작용을 피하고, 중재자를 통해 상호작용하도록 합니다.
    • 구현 : 중재자 객체는 객체들 간의 상호작용을 조정하고, 객체들은 중재자를 통해 상호작용합니다.
  6. 옵서버(Observer) 패턴
    • 목적 : 객체 사이에 일대다 종속 관계를 정의하여, 객체의 상태 변화를 다른 객체에 자동으로 알립니다.
    • 구현 : 주제(subject) 객체는 옵서버(observer) 객체들의 목록을 유지하고, 상태 변화가 있을 때 옵서버들에게 알립니다.
  7. 상태(State) 패턴
    • 목적 : 객체의 내부 상태에 따라 동작을 변경하고, 객체의 상태 전이를 관리합니다.
    • 구현 : 상태 객체들을 정의하고, 객체는 현재 상태에 따라 행동을 수행합니다.
  8. 전략(Strategy) 패턴
    • 목적 : 알고리즘을 캡슐화하고, 동적으로 알고리즘을 교체할 수 있게 합니다.
    • 구현 : 전략 객체를 정의하고, 컨텍스트 객체는 전략 객체를 사용하여 특정 알고리즘을 실행합니다.
  9. 템플릿 메서드(Template Method) 패턴
    • 목적 : 알고리즘의 구조를 정의하고, 일부 단계를 서브클래스에서 구현할 수 있게 합니다.
    • 구현 : 추상 클래스에서 알고리즘의 구조를 정의하고, 서브클래스에서 특정 단계를 구현합니다.
  10. 방문자(Visitor) 패턴
    • 목적 : 객체의 구조와 기능을 분리하여 새로운 기능을 추가할 수 있게 합니다.
    • 구현 : 방문자 객체는 객체 구조를 순회하면서 특정 기능을 수행하고, 각 객체는 방문자를 받아들입니다.
  11. 메멘토(Memento) 패턴
    • 목적 : 객체의 상태를 저장하고 이전 상태로 복원할 수 있는 방법을 제공합니다.
    • 구현 : 오리진터 객체는 상태를 저장하고 복원하는 메멘토 객체를 생성하고 사용합니다.

행위 패턴은 객체들 간의 상호작용과 책임 분배에 관련된 디자인 패턴입니다. 이러한 패턴은 객체들이 어떻게 상호작용하고 정보를 교환하며, 책임을 분산시키는 방법을 제공하여 소프트웨어 시스템을 유연하게 만듭니다.
